# Releasing

This guide describes how a maintainer cuts a release of agent-gpu. Releases are produced
automatically by [GoReleaser](https://goreleaser.com) when a version tag is pushed — the
process is reproducible and requires no manual artifact building.

## What a release produces

Pushing a tag matching `v*` triggers the
[`release.yml`](../.github/workflows/release.yml) workflow, which runs GoReleaser
(config: [`.goreleaser.yaml`](../.goreleaser.yaml)) and publishes a GitHub release containing:

- Six binary archives — Windows, macOS, and Linux, each for `amd64` and `arm64`
  (`.tar.gz` for macOS/Linux, `.zip` for Windows). Each archive bundles `LICENSE`,
  `README.md`, and `CHANGELOG.md`.
- `checksums.txt` — a SHA-256 manifest covering every archive.
- Release notes generated by GitHub's native generator, which honors the categories in
  [`.github/release.yml`](../.github/release.yml).

The binaries embed their version metadata via `-ldflags -X`, so `agentgpu --version` reports
the released version, the commit, and the build date.

## Cutting a release

1. Make sure `main` is green in CI and you are on an up-to-date checkout of `main`.
2. Update [`CHANGELOG.md`](../CHANGELOG.md): rename the `[Unreleased]` section to the new
   version with today's date, and add a fresh empty `[Unreleased]` section above it. Update the
   link footer.
3. Commit the changelog on a branch and merge it through the normal PR flow.
4. From the merged `main`, create and push an annotated tag:

   ```bash
   git checkout main
   git pull
   git tag -a v0.1.0 -m "v0.1.0"
   git push origin v0.1.0
   ```

5. The `Release` workflow runs GoReleaser, which cross-compiles all six targets, archives them,
   writes `checksums.txt`, and creates the GitHub release. Watch the workflow run and confirm
   the release and its artifacts appear on the
   [Releases page](https://github.com/jaypetez/agent-gpu/releases).

Versions follow [Semantic Versioning](https://semver.org/spec/v2.0.0.html). The project is
pre-1.0, so the first tag is `v0.1.0`; pre-release tags (e.g. `v0.1.0-rc1`) are automatically
marked as pre-releases.

## Reproducible builds

The build is deterministic so anyone can reproduce a published binary:

- `CGO_ENABLED=0` — fully static binaries with no C dependencies (the project deliberately
  avoids cgo; see [docs/architecture.md](architecture.md)).
- `-trimpath` — strips local filesystem paths from the binary.
- `-ldflags "-s -w"` — omits the symbol table and debug info.
- `mod_timestamp` pinned to the commit timestamp — stable file modification times.

## Testing the pipeline locally

You do not need to push a tag to validate the release configuration. The same checks run on
every pull request via the `release-dryrun` job in [`ci.yml`](../.github/workflows/ci.yml),
and you can run them locally:

```bash
make release-check   # validate .goreleaser.yaml
make snapshot        # cross-compile all six targets into dist/ (no publish)
```

`make snapshot` writes the binaries under `dist/` (git-ignored). Inspect them to confirm all
six OS/arch combinations were produced before tagging.
